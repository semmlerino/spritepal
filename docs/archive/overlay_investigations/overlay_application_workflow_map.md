1. Tile lifecycle and data flow  
- ROM → editor state: `ROMWorkflowController` holds `current_tile_data` (raw 4bpp bytes) and `current_width/current_height`; the arrangement dialog is opened from that state and renders a temp PNG via `SpriteRenderer.render_4bpp` in `ROMWorkflowController.show_arrangement_dialog` `ui/sprite_editor/controllers/rom_workflow_controller.py:1158`  
- Temp PNG → tiles dict: `GridImageProcessor.process_sprite_sheet_as_grid` loads the PNG as grayscale and splits it into tiles stored in `self.tiles: dict[TilePosition, Image.Image]` keyed by `(row, col)` in the physical grid `ui/row_arrangement/grid_image_processor.py:324` and `ui/grid_arrangement_dialog.py:90`  
- Arrangement workspace: `GridArrangementManager` tracks a canvas mapping `grid_mapping[(canvas_r, canvas_c)] = (ArrangementType, key)` where tile keys are `"row,col"` from the source grid, while `_arrangement_order` and `_arranged_tiles` are derived/maintained separately `ui/row_arrangement/grid_arrangement_manager.py:111`  
- Tile movement: drag/drop on the arrangement canvas mutates `grid_mapping` via `CanvasPlaceItemsCommand` and `CanvasMoveItemsCommand`, which call `set_item_at`/`move_grid_item` and re-derive arrangement order row-major from grid positions `ui/grid_arrangement_dialog.py:1111`, `ui/row_arrangement/undo_redo.py:496`, `ui/row_arrangement/grid_arrangement_manager.py:140`  
- Overlay modifies tile pixels only: `ApplyOverlayCommand` updates `self.tiles` for `TilePosition` keys without changing tile identities or order `ui/row_arrangement/undo_redo.py:612`  
- Dialog → ROM workflow: on accept, `ArrangementResult.modified_tiles` (a copy of `self.tiles`) is passed back; `ROMWorkflowController._update_tile_data_from_modified_tiles` patches `current_tile_data` by `tile_idx = row * tiles_per_row + col` and `offset = tile_idx * 32` `ui/grid_arrangement_dialog.py:1622`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1245`  
- Editor view vs ROM layout: `open_in_editor` applies `ArrangementBridge.physical_to_logical` for display if arrangement is kept, while `save_to_rom` reverses with `logical_to_physical` before injection, preserving ROM ordering `ui/sprite_editor/controllers/rom_workflow_controller.py:889`, `ui/sprite_editor/services/arrangement_bridge.py:238`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1757`  

Ambiguity / risk: the mapping back to ROM assumes `tiles_per_row = current_width // 8` reflects the physical ROM tile stride; the flow does not show any non-contiguous ROM offset mapping beyond row-major indexing (`ui/sprite_editor/controllers/rom_workflow_controller.py:1158`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1245`).

2. Overlay application behavior  
- Overlay layer state: `OverlayLayer` stores image, position `(x,y)`, scale, opacity, visibility; import emits `image_changed`, `scale_changed`, `position_changed` and auto-shows the overlay `ui/row_arrangement/overlay_layer.py:84`  
- Canvas overlay positioning: `OverlayGraphicsItem.itemChange` syncs drag movement back to `OverlayLayer.set_position`, which emits `position_changed` `ui/row_arrangement/overlay_item.py:56`  
- Apply flow: `GridArrangementDialog._apply_overlay` builds an `ApplyOperation` with `overlay`, `grid_mapping`, `tiles`, and optional palette; it validates and then executes to generate `modified_tiles` keyed by `TilePosition` derived from the tile key (`"row,col"`) `ui/grid_arrangement_dialog.py:848`, `core/apply_operation.py:49`  
- Sampling logic: `ApplyOperation.execute` iterates canvas positions `(r,c)` from `grid_mapping`, computes canvas pixel coordinates `(tile_x, tile_y)` from canvas positions, samples `OverlayLayer.sample_region`, and writes back grayscale or palette-quantized pixels to the tile identity extracted from the key `core/apply_operation.py:143`, `ui/row_arrangement/overlay_layer.py:280`  
- Arrangement dependence: overlay sampling uses **canvas positions** `(r,c)` while tile identity comes from the tile key. The arrangement position affects which overlay pixels are applied, but not which tile identity is modified.  

Potential risk: if arranged tiles exist only in `_arrangement_order` or groups (no `grid_mapping` entries), `_apply_overlay` allows execution because `get_arranged_tiles()` is non-empty, but `ApplyOperation` iterates an empty `grid_mapping` and modifies zero tiles (`ui/grid_arrangement_dialog.py:848`, `ui/row_arrangement/grid_arrangement_manager.py:259`).

3. Post-overlay expectations  
- Tile identities remain unchanged: `ApplyOperation` uses the tile key from `grid_mapping` (`"row,col"`) as the identity, and `ApplyOverlayCommand` only replaces images for those keys `core/apply_operation.py:143`, `ui/row_arrangement/undo_redo.py:612`  
- ROM ordering remains unchanged: `logical_to_physical` reverses any arrangement before injection, and `_update_tile_data_from_modified_tiles` patches by the original row-major order `ui/sprite_editor/services/arrangement_bridge.py:305`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1245`  
- Only pixel data is modified: no changes to tile indices, mapping, or ordering; updates are confined to `self.tiles` and `current_tile_data` byte offsets `ui/row_arrangement/undo_redo.py:612`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1245`  
- Pixel editor reflects final data: if `modified_tiles` is included in the accepted result, `current_tile_data` is patched and `open_in_editor` reloads from it; then save reverses arrangement for ROM injection `ui/sprite_editor/controllers/rom_workflow_controller.py:1158`, `ui/sprite_editor/controllers/rom_workflow_controller.py:889`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1757`  

Potential inconsistencies / risks:  
- Applying overlay and then changing arrangement clears `_apply_result`, so `modified_tiles` is omitted from `ArrangementResult`; the dialog still shows modified tiles, but the changes will not propagate to `current_tile_data` on accept `ui/grid_arrangement_dialog.py:993`, `ui/grid_arrangement_dialog.py:1622`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1158`  
- The source grid display is derived from `original_image`, not `self.tiles`; after overlay, the source grid can remain visually stale while the arrangement canvas reflects changes `ui/grid_arrangement_dialog.py:1230`, `ui/grid_arrangement_dialog.py:1314`  

4. UI consistency  
- “Keep layout in editor after closing” toggles whether the arrangement bridge is applied in the editor; overlay pixel changes can still be applied even when layout is discarded (`keep_arrangement` only affects `_current_arrangement`) `ui/grid_arrangement_dialog.py:1658`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1158`  
- “Apply Overlay to Arranged Tiles” applies only to tiles present in the canvas mapping; it does **not** reindex tiles, only replaces pixels for those tile identities `ui/grid_arrangement_dialog.py:848`, `core/apply_operation.py:143`  
- The overlay is auto-hidden after apply, which disables the apply button until the overlay is re-shown `ui/grid_arrangement_dialog.py:941`, `ui/grid_arrangement_dialog.py:1004`  
- Arrangement canvas tiles reflect `self.tiles` (including overlay changes), while the left source grid uses `original_image` — a potential visual mismatch after overlay `ui/grid_arrangement_dialog.py:1314`, `ui/grid_arrangement_dialog.py:1230`  

5. Signal and event flow (tile rearrangement, overlay, pixel updates, reinjection)  
Tile rearrangement  
- `GridGraphicsView.tiles_dropped` → `GridArrangementDialog._on_tiles_dropped_on_canvas` → `CanvasPlaceItemsCommand` or `CanvasMoveItemsCommand` → `GridArrangementManager.set_item_at/move_grid_item` → `arrangement_changed.emit` → `GridArrangementDialog._on_arrangement_changed` → `_update_displays` `ui/components/visualization/grid_graphics_view.py:60`, `ui/grid_arrangement_dialog.py:1111`, `ui/row_arrangement/undo_redo.py:496`, `ui/row_arrangement/grid_arrangement_manager.py:140`, `ui/grid_arrangement_dialog.py:993`  
- `GridGraphicsView.tile_clicked` and `tiles_selected` update selection state; they do not alter arrangement state directly `ui/components/visualization/grid_graphics_view.py:60`, `ui/grid_arrangement_dialog.py:379`  
- Signals emitted more than once: bulk operations (`AddMultipleTilesCommand`, multi-tile drag) call `_add_tile_no_history` or `move_grid_item` per tile, and each call emits `arrangement_changed`, causing repeated UI refreshes `ui/row_arrangement/undo_redo.py:232`, `ui/row_arrangement/grid_arrangement_manager.py:557`  

Overlay application  
- `OverlayControls` actions → `OverlayLayer.import_image/set_position/set_scale/set_opacity/set_visible` → emits `image_changed/position_changed/scale_changed/opacity_changed/visibility_changed` → `GridArrangementDialog._on_overlay_changed` (and also `OverlayControls` UI updates) `ui/row_arrangement/overlay_controls.py:127`, `ui/row_arrangement/overlay_layer.py:84`, `ui/grid_arrangement_dialog.py:1004`  
- `OverlayGraphicsItem.itemChange` (drag) → `OverlayLayer.set_position` → `position_changed` → `_on_overlay_changed` `ui/row_arrangement/overlay_item.py:56`, `ui/grid_arrangement_dialog.py:1004`  
- `apply_overlay_btn.clicked` → `_apply_overlay` → `ApplyOperation.execute` → `ApplyOverlayCommand.execute` → tiles dict updated → `_update_displays` callback → overlay auto-hidden (visibility signal) `ui/grid_arrangement_dialog.py:848`, `core/apply_operation.py:143`, `ui/row_arrangement/undo_redo.py:612`  
- Missing listeners: `GridArrangementManager.tile_added`, `tile_removed`, `group_added`, `group_removed`, `arrangement_cleared` are emitted but not connected in the dialog flow `ui/row_arrangement/grid_arrangement_manager.py:54`  

Pixel data updates and ROM reinjection  
- Dialog accept → `ArrangementResult.modified_tiles` → `_update_tile_data_from_modified_tiles` patches `current_tile_data` (no signals emitted) `ui/grid_arrangement_dialog.py:1622`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1245`  
- UI trigger: `EditWorkspace.saveToRomRequested` → `ROMWorkflowController.prepare_injection` → `save_to_rom` → `rom_extractor.inject_sprite_to_rom` → UI updates and `set_offset` for refresh `ui/sprite_editor/controllers/rom_workflow_controller.py:292`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1667`, `ui/sprite_editor/controllers/rom_workflow_controller.py:1757`  

Ordering dependency: `GridArrangementDialog._on_arrangement_changed` clears `_apply_result` (unless `_is_applying_overlay`), so any arrangement modification after apply invalidates the stored overlay result even though tile pixels remain modified in-memory `ui/grid_arrangement_dialog.py:993`.
