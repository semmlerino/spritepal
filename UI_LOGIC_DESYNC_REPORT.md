# Audit Report: UI–Logic Desynchronization
**Project:** SpritePal  
**Date:** January 11, 2026  
**Scope:** Sprite Editor Workflow (Extraction, Asset Browsing, Modification, Injection)

## Summary
The audit identified four significant areas where the UI state fails to accurately reflect the underlying logic or model state. The highest risk is associated with the **Undo/Redo system**, where a model reset (loading a new image) leaves the UI in a stale, "operable" state that can lead to crashes or data corruption.

## Findings

### 1. The Undo State Signal Gap (High Risk)
*   **Source of Truth:** `EditingController.undo_manager`.
*   **Expected UI Reflection:** `MainWindow.undo_action` (Undo button) must be disabled when a new image is loaded.
*   **Failure Mode:** Missing Signal. `load_image` clears the logic stack but fails to emit `undoStateChanged`.
*   **Impact:** Triggering "Undo" on a stale UI state after a model reset can cause index out-of-bounds errors or logical corruption.
*   **Evidence:** `ui/sprite_editor/controllers/editing_controller.py:L142` calls `self.undo_manager.clear()` but no signal is emitted.

### 2. Global Toolbar Offset "Ghosting" (Medium Risk)
*   **Source of Truth:** `ROMExtractionPanel.manual_offset`.
*   **Expected UI Reflection:** `MainWindow.sprite_preview` should update when a ROM offset is entered via the toolbar.
*   **Failure Mode:** Stale UI / Partial Update. The offset is synced to internal logic, but the visual preview is only triggered for VRAM offsets.
*   **Impact:** User enters a valid ROM offset but sees no visual confirmation, making the tool appear unresponsive.
*   **Evidence:** `ui/main_window.py:L268` (`_on_global_offset_changed`) navigates but does not trigger a preview update in ROM mode.

### 3. Editor Workspace State Divergence (Medium Risk)
*   **Source of Truth:** `EditWorkspace` internal state (`_zoom`, `_grid_visible`).
*   **Expected UI Reflection:** Zoom and Grid settings should persist when switching between VRAM and ROM workflow modes.
*   **Failure Mode:** UI reflects internal state instead of public contract. The "Editor" uses two independent sets of widgets for different modes.
*   **Impact:** UX friction; users lose configuration/zoom level when switching modes.
*   **Evidence:** `ui/workspaces/sprite_editor_workspace.py` creates distinct `EditWorkspace` instances for VRAM and ROM pages.

### 4. ROM Extraction Result Path Desync (Low Risk)
*   **Source of Truth:** `MainWindow._output_path`.
*   **Expected UI Reflection:** `OutputSettingsManager` labels should reflect the actual file generated by the ROM worker.
*   **Failure Mode:** Partial Update. The worker uses the dialog-specified name, but the manager doesn't sync this back to the main UI.
*   **Impact:** Minor UI inconsistency; the "Output Name" field may display a different value than the last successfully extracted file.
*   **Evidence:** `ui/main_window.py:L1173` (`_on_rom_extraction_finished`) updates the path but not the manager's UI state.

## Addendum: Recommendations

### R1. Fix Undo Signal Emission (Urgent)
*   Modify `EditingController.load_image` to call `self._emit_undo_state()` immediately after clearing the undo manager.
*   Add a regression test in `tests/ui/controllers/test_editing_controller_undo_desync.py` to assert that `undoStateChanged(False, False)` is emitted.

### R2. Unified Preview Logic
*   Implement `_update_rom_preview` in `MainWindow` similar to `_update_preview_with_offset`.
*   Connect `rom_extraction_panel.manual_offset_changed` to this new method to ensure real-time preview updates for ROM offsets.

### R3. Centralize Editor Viewport State
*   Refactor `PixelCanvas` zoom and toggle states (Grid, Tile Grid, Greyscale) into the `EditingController`.
*   Ensure that when an `EditWorkspace` is initialized or switched, it pulls these values from the controller.

### R4. Synchronize Output Manager
*   Update `MainWindow.extraction_complete` to call `self.output_settings_manager.set_output_name(actual_name)` to ensure the UI matches the file system state.

### R5. Test Coverage Expansion
*   Implement explicit UI integration tests that assert **public signal emission** rather than just internal model changes.
*   Use `qtbot.wait_signal` to verify that UI-triggering actions (like loading an image) result in the expected state-change notifications.
